# RAG From Scratch — Basic to Advanced

A hands-on implementation of Retrieval-Augmented Generation (RAG) techniques, built progressively from a simple pipeline to advanced strategies. Each folder is a standalone, runnable implementation.

## What is RAG?

Large Language Models (LLMs) are trained on fixed data and don't know about your private documents or recent events. RAG solves this by:

1. **Indexing** — loading your documents, splitting them into chunks, embedding them, and storing in a vector database
2. **Retrieval** — when a question comes in, finding the most relevant chunks
3. **Generation** — feeding those chunks as context to the LLM to generate a grounded answer

---

## Tech Stack

| Component | Tool |
|---|---|
| LLM | Groq (`llama-3.1-8b-instant`) — free API |
| Embeddings | HuggingFace (`all-MiniLM-L6-v2`) — runs locally, free |
| Vector Database | ChromaDB — local, no server needed |
| PDF Loader | LangChain `PyPDFDirectoryLoader` |
| Web Search | DuckDuckGo — free, no API key |
| Framework | LangChain |

## Setup

**1. Clone the repo**
```bash
git clone https://github.com/moreshivam/RAG-.git
cd RAG-
```

**2. Install dependencies**
```bash
pip install -r requirements.txt
```

**3. Add your API key**

Create a `.env` file in the root:
```
GROQ_API_KEY=your_groq_api_key_here
```
Get a free Groq key at [console.groq.com](https://console.groq.com)

**4. Add your PDFs**

Drop any PDF files into the `data/` folder.

**5. Index your documents (run once)**
```bash
cd 01_basic_rag
python ingest.py
```

## Techniques

### 01 — Basic RAG
The foundation. Load PDFs → split into chunks → embed → store in ChromaDB. On query: embed question → retrieve top-k chunks → feed to LLM.

```bash
cd 01_basic_rag && python rag.py
```

**LLM calls per question:** 1

---

### 02 — Multi-Query
Generates 3 alternative phrasings of your question, retrieves chunks for each, merges and deduplicates results. Wider retrieval coverage than basic RAG.

```
"What is RAG?" → 3 versions → 3 retrievals → merged chunks → answer
```

```bash
cd 02_multi_query && python rag.py
```

**LLM calls per question:** 2 (1 for query generation + 1 for answer)

---

### 03 — RAG Fusion
Builds on Multi-Query by adding **Reciprocal Rank Fusion (RRF)** scoring. Chunks that appear consistently at the top across multiple queries get boosted. Smarter than naive merging.

```
RRF score = Σ 1 / (60 + rank)
```

```bash
cd 03_rag_fusion && python rag.py
```

**LLM calls per question:** 2

---

### 04 — Combined HyDE + RAG Fusion
**HyDE** (Hypothetical Document Embeddings): instead of embedding the question, generates a hypothetical answer and embeds that. Hypothetical answers match document chunks better in embedding space.

Combined with RAG Fusion for maximum retrieval quality:
- Generate 3 alternative queries
- For each query → generate a hypothetical answer
- Embed hypothetical answers → search vectorstore
- RRF rank all results

```bash
cd 04_combined_fusion_hyde && python rag.py
```

**LLM calls per question:** 5 (1 query gen + 3 hypothetical answers + 1 final answer)

---

### 06 — Multi-Representation Indexing
Separates **what you search on** from **what you retrieve**.

- Stores short LLM-generated summaries in vectorstore (precise search)
- Stores full document pages in a separate docstore (rich context)
- Summaries and full docs linked via shared IDs

```
Search on summaries → retrieve full pages → LLM gets rich context
```

```bash
cd 06_multi_representation
python ingest.py   # builds both stores (run once)
python rag.py
```

**LLM calls for indexing:** 1 per document page (for summary generation)

---

### 07 — CRAG (Corrective RAG)
Adds a **quality check** after retrieval. Grades each retrieved chunk as relevant or not. Falls back to web search if no relevant chunks found.

```
retrieve → grade each chunk → relevant? use them : web search
```

```bash
cd 07_crag && python rag.py
```

**LLM calls per question:** 1 per chunk (grading) + 1 for answer

---

### 08 — Adaptive RAG + CRAG
The most complete pipeline. Decides the retrieval **strategy** before retrieving, then validates with CRAG.

**Adaptive RAG** routes to one of three strategies:
- `simple` → basic retrieval
- `complex` → multi-query retrieval
- `web_search` → skip vectorstore, go straight to web

**CRAG** then grades the retrieved chunks and falls back to web search if needed.

```
question → classify → retrieve (right strategy) → grade → answer
```

```bash
cd 08_adaptive_rag && python rag.py
```

## Technique Comparison

| Technique | Key Idea | LLM Calls | Best For |
|---|---|---|---|
| Basic RAG | Single retrieval | 1 | Simple, fast queries |
| Multi-Query | Multiple query versions | 2 | When phrasing varies |
| RAG Fusion | Multiple queries + RRF ranking | 2 | Better ranking of results |
| HyDE + Fusion | Hypothetical answers + RRF | 5 | Best retrieval quality |
| Multi-Representation | Summary index + full doc retrieval | 1/page (indexing) | Long, dense documents |
| CRAG | Self-correcting retrieval | varies | Unreliable document coverage |
| Adaptive + CRAG | Strategy selection + correction | varies | Production-ready pipeline |

## Learning Path

Follow the folders in order — each builds on the previous:

```
01 Basic RAG → understand the core loop
      ↓
02 Multi-Query → better retrieval coverage
      ↓
03 RAG Fusion → smarter ranking
      ↓
04 HyDE + Fusion → best retrieval quality
      ↓
06 Multi-Representation → smarter indexing
      ↓
07 CRAG → self-correcting
      ↓
08 Adaptive + CRAG → full production pipeline
```
